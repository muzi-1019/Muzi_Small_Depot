# 基于 RAG 的角色扮演系统 — 项目问答总结

---

## 一、项目概述

### 1. 项目名称与定位

**项目名称**：基于 RAG 的角色扮演系统（RAG Role-Play System）

**项目定位**：类似 Character.ai，构建一个可扮演多种角色（医生、律师、虚拟朋友等）的智能聊天机器人。系统结合 **大语言模型（LLM）** 与 **RAG（Retrieval-Augmented Generation，检索增强生成）** 技术，让每个角色拥有专属知识库，回答更专业、更精准。

### 2. 支持的角色类型

| 领域 | 角色示例 | 数据来源 |
|------|---------|---------|
| 社交 | 虚拟朋友（NPC） | 自定义人设与提示词 |
| 医疗 | 高血压专科医生、中医、心理医生 | 国家卫健委《高血压治疗指南》 |
| 法律 | 民事律师 | 法律条文、司法解释、中国裁判文书网判例 |
| 金融 | 股票/证券投资顾问、金融理财师 | 东方财富网数据 |
| 教育 | 教师、英语学习助手 | 教材、题库 |
| 科学 | 科学家 | 论文、百科 |

系统中已预置 3 个角色：**虚拟朋友**、**高血压专科医生**、**民事律师**。可通过数据库 `character` 表自行扩展。

### 3. NPC（Non Player Character）

NPC 即"非玩家角色"，原意指游戏中由系统控制的虚拟角色。在本项目中，NPC 指系统扮演的虚拟人物，拥有自己的**人物性格（persona）**、**领域知识**和**提示词模板**，用户与之对话时，系统会按照该角色的设定来回答。

## 二、需求分析

### 1. 产出物：需求规格说明书

本项目的需求规格说明书位于 `docs/需求规格说明书.md`，包含以下核心内容：

- **功能需求**：用户注册/登录、角色管理、多轮对话、知识库上传与检索、会话管理（创建/删除/重命名）
- **业务流程**：用户注册 → 登录获取 JWT → 选择角色 → 创建会话 → 发送消息 → 系统检索知识库 + 读取记忆 → 调用大模型生成回复 → 存储消息与记忆
- **业务规则**：
  - 每个用户最多同时与 3 个角色保持活跃对话
  - 超过空闲时间（默认 3600 秒）未发言的角色自动释放名额
  - 密码使用单向哈希存储（pbkdf2_sha256），不可逆
  - 知识库仅支持 txt/pdf/md 格式

### 2. 业务流程图（文字描述）

```
用户 ──注册──→ MySQL(user 表)
  │
  ├──登录──→ 验证密码 → 签发 JWT → 返回 token + user_id
  │
  ├──选择角色──→ 读取 character 表
  │
  ├──创建/选择会话──→ MySQL(conversation 表)
  │
  ├──发送消息──→ ChatService 编排：
  │     ├─ 1. 检查并发角色槽位（Redis）
  │     ├─ 2. 从 Milvus 检索知识片段（RAG）
  │     ├─ 3. 从 Redis 读取本会话短期记忆
  │     ├─ 4. 拼接系统提示词 + 知识 + 记忆 + 用户问题
  │     ├─ 5. 调用大模型生成回答
  │     ├─ 6. 消息写入 MySQL(chat_message 表)
  │     └─ 7. 记忆追加到 Redis
  │
  └──上传知识文件──→ 保存到 uploads/ → 元数据写入 MySQL → PDF 解析切分 → 向量化写入 Milvus
```

## 三、技术架构

### 1. 总体架构

```
[React 前端 / Postman / JMeter]
        │  HTTP JSON
        ▼
[FastAPI 后端服务（单体）]
   ├─ 用户与鉴权        → MySQL: user 表
   ├─ 角色元数据        → MySQL: character 表
   ├─ 会话与消息        → MySQL: conversation, chat_message 表
   ├─ 知识库元数据      → MySQL: knowledge_document 表 + 本地 uploads/
   ├─ 短期记忆         → Redis: List + Hash
   ├─ RAG 检索引擎     → Milvus 向量检索
   └─ 大模型网关       → OpenAI 兼容 API（本地部署或在线服务）
```

| **大模型** | DeepSeek-V3（硅基流动 API） | 生成对话回复 |

- **MySQL**：存储用户信息、角色信息、会话记录、知识文档元数据
- **Redis**：短期记忆存储、并发角色控制
- **Milvus**：向量检索引擎，支持混合检索
- **BGE-m3**：文本向量化模型（Embedding Model）
- **BGE-rerank**：重排序模型
- **RAGAS**：RAG 系统评测框架

| `prompt_template` | 提示词模板 | 高血压医生模板 |
| `knowledge_base_id` | 关联的知识库分区 | medical_hypertension |

### 2. 提示词模板机制

系统在调用大模型时，会将角色信息组装成 system prompt：

```
你是「高血压专科医生」，领域：医疗。
人设：严谨、专业、注重风险提示
提示模板：高血压医生模板

【检索到的知识片段】
（来自 Milvus 的检索结果）

【近期对话记忆】
（来自 Redis 的短期记忆）

【用户问题】
高血压日常饮食要注意什么？
```

这种方式使每个角色的回答风格和知识范围各不相同。

### 3. 知识库（Milvus）

每个角色拥有独立的知识库分区（通过 `character_id` 隔离）：

- **高血压医生** → 国家基层高血压防治管理手册
- **民事律师** → 中华人民共和国宪法及相关法律条文

知识库存储在 Milvus 的 `character_knowledge` Collection 中。

## 六、向量化与检索

### 1. 向量化模型：BGE-m3

BGE-m3 是 BAAI（智源研究院）开源的多功能嵌入模型，支持：

- **稠密向量**（Dense）：语义相似度搜索
- **稀疏向量**（Sparse）：类似 BM25 的关键词匹配
- **多语言**：中英文等多语言支持

本项目配置向量维度为 768 维（`RAG_MILVUS_DIM=768`）。

### 2. 重排序模型：BGE-rerank

检索到候选文档后，使用 BGE-rerank 对结果重新排序，筛选出最相关的 Top-K 片段（默认 `RAG_RERANK_TOP_K=5`）。

### 3. Milvus 混合检索

Milvus Collection 设计：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INT64（主键，自增） | 唯一标识 |
| `character_id` | INT64 | 角色 ID，用于分区过滤 |
| `source_file` | VARCHAR(255) | 文档来源文件名 |
| `chunk_hash` | VARCHAR(64) | 文本块的 SHA256 哈希（用于去重） |
| `text` | VARCHAR(65535) | 向量对应的原文（支持 BM25 混合检索） |
| `vector` | FLOAT_VECTOR(768) | 文本的稠密向量表示 |

**混合检索流程**：稠密向量 ANN 搜索 + BM25 稀疏检索 → 合并结果 → BGE-rerank 重排序 → 返回 Top-K。

## 七、多轮对话与记忆机制

### 1. 核心问题

大模型本身**没有记忆**——每次调用都是独立的，不知道之前聊了什么。因此需要外部记忆系统。

### 2. 短期记忆（Redis）

- **存储结构**：Redis List
- **Key 格式**：`chat:session:{user_id}:{character_id}:{conversation_id}`
- **内容**：最近 N 轮对话（默认 20 轮，即 40 条消息）
- **操作**：`RPUSH` 追加 + `LTRIM` 裁剪，保证 O(1) 常数时间复杂度

**Redis 5 种常用数据类型**：

| 类型 | 本项目用途 |
|------|-----------|
| **String** | 可用于缓存单个值（如配置） |
| **List** | 存储短期对话记忆（有序、可裁剪） |
| **Set** | 可用于去重场景 |
| **Hash** | 存储并发角色活跃状态（`chat:active_roles:{user_id}`） |
| **Sorted Set（ZSet）** | 可用于按分数排序的排行榜场景 |

本项目主要使用了 **List**（短期记忆）和 **Hash**（并发角色控制）。

Redis 的核心优势：**O(1) 常数时间复杂度**的键值操作，读写极快，适合做缓存和实时数据存储。

### 3. 长期记忆（Milvus）

长期记忆可通过 Milvus 向量库实现：将重要的对话摘要向量化后存入 Milvus，在后续对话中通过语义检索召回相关历史。

### 4. 会话隔离

每个会话（conversation）拥有独立的记忆空间，会话 A 的聊天内容不会泄露到会话 B。通过在 Redis key 中加入 `conversation_id` 实现。

## 八、多用户、多角色

### 1. 多用户

- 每个用户有独立的账号、密码、JWT 令牌
- 用户之间的会话、消息、记忆完全隔离
- 通过 `user_id` 进行数据隔离

### 2. 多角色

- 系统预置多个角色，用户可自由切换
- 每个用户最多**同时**与 3 个角色保持活跃对话
- 通过 Redis Hash `chat:active_roles:{user_id}` 记录每个角色的最近活跃时间
- 空闲超过 `RAG_ACTIVE_ROLE_IDLE_SECONDS`（默认 3600 秒）的角色自动释放名额

## 九、知识库动态更新

### 1. 上传流程

1. 用户通过前端上传 txt / pdf / md 文件
2. 后端校验文件类型 → 保存到 `uploads/{character_id}/{uuid}.ext`
3. 元数据写入 MySQL `knowledge_document` 表（状态：`pending`）
4. 如果是 PDF：
   - 使用 pypdf 提取文本
   - 固定长度分块（800 字符，120 字符重叠）
   - 向量化后写入 Milvus
   - 更新状态为 `processed` 或 `failed`

### 2. PDF 处理技术

| 工具 | 用途 |
|------|------|
| **PyMuPDF（fitz）** | 高性能 PDF 文本提取 |
| **pypdf** | 纯 Python PDF 文本提取（本项目使用） |
| **PDFPlumber** | 擅长提取表格数据 |
| **PaddleOCR** | 图片文字识别（OCR） |
| **多模态大模型** | 理解图片中的文字和图表 |
| **MinerU** | 文档智能解析 |
| **Paddle-OCR-VL** | 视觉语言 OCR |

## 十二、RAG 完整流程

### 1. 离线阶段（知识入库）

```
PDF/TXT/MD 文件
    → 文本提取（pypdf / PyMuPDF）
    → 文本清洗（去除多余空白）
    → 文本分块（固定长度 800 字符，重叠 120 字符）
    → BGE-m3 向量化（稠密 + 稀疏向量）
    → 写入 Milvus Collection
```

### 2. 在线阶段（对话检索）

```
用户提问
    → Query 向量化（BGE-m3）
    → Milvus 混合检索（向量 ANN + BM25）
    → BGE-rerank 重排序
    → 取 Top-K 知识片段
    → 拼接 system prompt + 知识 + 记忆 + 用户问题
    → 调用大模型生成回复
```

### 3. 分块策略

| 策略 | 说明 |
|------|------|
| **固定长度** | 按字符数切分（本项目使用：800 字符，120 字符重叠） |
| **按句子** | 以句号、问号等断句 |
| **按段落** | 以空行分隔 |
| **按标题** | 以 Markdown 标题层级切分 |
| **语义分块** | 根据语义相似度动态切分 |

## 十四、RAG 评测：RAGAS

### 1. 什么是 RAGAS

RAGAS（Retrieval Augmented Generation Assessment）是一个专门评估 RAG 系统效果的开源框架。

### 2. 评测指标

| 指标 | 含义 |
|------|------|
| `context_precision` | 检索片段中与问题相关的比例 |
| `context_recall` | 是否覆盖参考答案中的关键信息 |
| `faithfulness` | 回答是否忠实于给定上下文（不编造） |
| `answer_relevancy` | 回答与用户问题的相关性 |

### 3. 本项目评测流程

1. 准备问题列表文件（`questions.txt`）
2. 运行 `scripts/export_ragas_rows.py` 批量调用 `/api/chat`，收集问题、检索上下文、回答
3. 导出 `ragas_rows.jsonl`
4. 使用 RAGAS 库加载数据集，选择评测 LLM 与嵌入模型
5. 调用 `evaluate(dataset, metrics=[...])` 生成评测报告

## 十七、RAG 框架对比

| 框架 | 特点 |
|------|------|
| **LangChain** | 功能最全面的 RAG 框架，6 大功能模块：Models、Prompts、Indexes、Memory、Chains、Agents |
| **LlamaIndex** | 专注于数据索引和检索，适合构建知识库 |
| **Dify** | 低代码平台，可视化搭建 RAG 应用 |
| **LightRAG** | 轻量级 RAG 框架，基于图结构 |
| **LinerRAG** | 线性 RAG 框架，简化检索流程 |

本项目采用**自研实现**，不依赖上述框架，直接使用 pymilvus + httpx 调用大模型，更灵活可控。

## 十八、Milvus Collection 设计

```
Collection: character_knowledge
├── id          : INT64   (主键，自增 ID，保证唯一性)
├── character_id: INT64   (角色 ID，用于数据隔离)
├── source_file : VARCHAR (文档来源文件名)
├── chunk_hash  : VARCHAR (文本块 SHA256 哈希，用于去重)
├── text        : VARCHAR (向量对应的原文，支持 BM25 混合检索)
└── vector      : FLOAT_VECTOR(768) (BGE-m3 生成的稠密向量)
```

可扩展字段：创建时间、修改时间、摘要等。

## 二十二、设计文档总结

### 功能设计

1. **用户鉴权**：注册 → pbkdf2_sha256 哈希存储 → 登录 → JWT 签发 → 后续请求携带 Token
2. **对话编排**：角色槽位检查 → RAG 检索 → 读取记忆 → 大模型生成 → 消息落库 → 记忆更新
3. **知识库管理**：文件上传 → 文本提取 → 分块 → 向量化 → Milvus 入库
4. **会话管理**：创建 / 列表 / 删除 / 重命名，每个会话独立记忆隔离

### 技术架构

- **前端**：React SPA，通过 HTTP JSON 与后端通信
- **后端**：FastAPI 单体服务，分层架构（API → Service → Repository）
- **存储**：MySQL（结构化数据） + Redis（缓存/记忆） + Milvus（向量检索）
- **AI**：LLM（在线 API 或本地部署） + BGE-m3（向量化） + BGE-rerank（重排序）
