# 系统设计文档（技术架构与功能设计）

## 1. 总体架构（文字拓扑）

```
[Web / Postman / JMeter] 
        │  HTTPS JSON
        ▼
[FastAPI 单体服务]
   ├─ 用户与偏好（MySQL: user）
   ├─ 角色元数据（MySQL: character）
   ├─ 会话与消息（MySQL: conversation, chat_message）
   ├─ 知识库元数据（MySQL: knowledge_document） + 本地文件 uploads/
   ├─ 短期记忆（Redis: chat:session:{user}:{role} 与 chat:active_roles:{user}）
   ├─ RAG 引擎（RAGService：可选 Milvus 查询 + mock 回退）
   └─ 大模型网关（LLMService：mock 或 OpenAI 兼容 HTTP）
```

后续可演进到 API Gateway + 独立对话服务 + 独立向量索引流水线，无需更改领域模型命名。

## 2. 技术栈

| 层级 | 技术 |
| --- | --- |
| Web 框架 | FastAPI |
| 关系库 | MySQL（SQLAlchemy ORM） |
| 缓存 | Redis |
| 向量库 | Milvus（可选，`pymilvus` 懒加载） |
| 嵌入/重排 | BGE-m3 / BGE-rerank（部署在独立推理服务，本仓库预留接口与配置位） |
| 大模型 | vLLM / SGLang / xInference（OpenAI 兼容）或在线 API |
| 评测 | RAGAS（离线脚本与说明文档） |

## 3. 核心模块

### 3.1 用户与鉴权

- `UserRepository`：注册、查询、偏好更新。
- `security`：`pbkdf2_sha256` 密码哈希；`python-jose` 签发 JWT。

### 3.2 对话编排（ChatService）

顺序：`ensure_concurrent_role_slot` → `RAGService.retrieve` → 读取 Redis 记忆 → `LLMService.generate` → MySQL 落库 → Redis 追加一轮。

### 3.3 记忆服务（MemoryService）

- List：`chat:session:{user_id}:{character_id}`，`LTRIM` 保留最近 `2 * max_rounds` 条字符串。
- 并发角色控制：`chat:active_roles:{user_id}` Hash，字段为 `character_id`，值为最近活跃时间戳。

### 3.4 知识库服务（KnowledgeService）

校验扩展名 → 写入 `uploads/{character_id}/{uuid}.ext` → 插入 `knowledge_document(status=pending)`。向量化与 Milvus upsert 建议在后台任务（Celery/RQ）实现。

### 3.5 RAGService

- `RAG_MILVUS_ENABLED=true` 时尝试 `connections.connect` + `Collection.query(expr character_id == ...)`。
- Collection 需至少包含标量字段 `character_id` 与 `text`（与代码示例一致，可按实际 schema 调整表达式与 output_fields）。
- 任意失败回退 mock，保证演示环境可启动。

### 3.6 LLMService

根据 `RAG_LLM_PROVIDER` 分支；OpenAI 兼容路径组装 system（人设+模板）与 user（检索片段+记忆+问题）。

## 4. 数据库设计（逻辑）

- `user`：账号凭证、昵称、`preferred_character_ids`（JSON 文本）。
- `character`：角色定义、提示词模板、`knowledge_base_id`。
- `conversation`：`(user_id, character_id)` 唯一会话（应用层 get_or_create），`updated_at` 在消息写入时刷新。
- `chat_message`：一问一答。
- `knowledge_document`：上传文件元数据与索引状态。

## 5. 配置项（环境变量前缀 `RAG_`）

详见仓库根目录 `.env.example`。关键项：`MYSQL_DSN`、`REDIS_URL`、`MILVUS_*`、`LLM_PROVIDER`、`OPENAI_API_BASE`、`OPENAI_API_KEY`、`SHORT_MEMORY_ROUNDS`、`MAX_CONCURRENT_ROLES_PER_USER`、`ACTIVE_ROLE_IDLE_SECONDS`、`UPLOAD_DIR`。

## 6. 风险与后续工作

- **Schema 演进**：已存在数据库需手工 `ALTER TABLE` 增加新列（`preferred_character_ids`、`updated_at`、`knowledge_document` 表等）。
- **Milvus 混合检索**：当前示例为标量过滤 + 文本字段查询占位；生产需接入 ANN 与稀疏向量字段及 rerank 服务。
- **异步索引**：上传后应触发队列任务解析 PDF、切分、调用嵌入服务写入 Milvus，并回写 `status=indexed/failed`。
