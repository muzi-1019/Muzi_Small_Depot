## 一、开篇：项目定位与整体架构（1分钟）

各位老师好，我今天介绍的这个项目叫 **“基于 RAG 的角色扮演对话系统”**。

简单来说，它的核心能力是：**让用户与不同的历史人物、虚拟角色进行对话，并且这些角色的回答不是凭空编造的，而是基于我上传的知识库（比如人物传记、专业文档）来生成的。**

从技术架构上看，系统采用**前后端分离**的设计。后端基于 **FastAPI** 框架，负责核心业务逻辑；前端基于 **React + Vite**，提供对话界面。整个代码组织遵循**分层架构**的原则，从上到下依次是：接口层、业务逻辑层、数据访问层、数据库层和基础配置层。

## 二、第一层：接口层 —— 系统的“大门”（1分钟）

**对应目录：`app/api/v1/`**

这一层是前后端交互的唯一入口。当你在网页上输入一句话、点击发送时，前端会发起一个 HTTP 请求，这个请求首先就会被接口层接收。

我们来看几个核心文件：

- **`chat.py`**：这是最核心的对话接口。它接收用户的消息、当前选择的角色ID，然后调用后端的业务逻辑，最后返回 AI 生成的回复。
- **`knowledge.py`**：负责知识库管理。当你上传一份 PDF 或 TXT 文件时，这个接口负责接收文件，并触发后续的向量化处理流程。
- **`characters.py`**：管理角色信息。比如创建“诸葛亮”这个角色，设定他的性格、背景，或者查询所有可用角色。
- **`auth.py`**：负责用户登录、注册、Token 校验，保证系统安全性。
- **`router.py`**：这是一个聚合文件，把所有接口统一注册到 FastAPI 应用上，相当于“接口的总目录”。

**这一层的作用可以概括为：** 接收请求、校验参数、调用下层服务、返回响应。它本身不处理复杂逻辑，相当于系统的“接待员”。

## 三、第二层：业务逻辑层 —— 系统的“大脑”（3分钟）

**对应目录：`app/services/`**

这一层是整个项目最核心的部分，所有智能化的能力都在这里实现。我重点讲五个关键服务：

### 1. `chat_service.py` —— 对话总控

这是整个流程的“总指挥”。当接口层把请求传下来后，它要做的事情是：

- 获取角色的系统提示词（比如“你扮演诸葛亮，你足智多谋、说话文雅”）；
- 从对话记忆里取出最近几轮的聊天记录，维持上下文连贯性；
- 调用 RAG 服务，检索与当前问题相关的知识片段；
- 把提示词、历史对话、检索到的知识拼接成一个完整的 Prompt；
- 最后调用大模型服务，生成回复。

### 2. `rag_service.py` —— 检索增强生成

这是 RAG 的核心。它的工作流程是：

- 把用户的问题转换成一个向量（embedding）；
- 去向量数据库中检索最相似的 Top-K 个知识片段；
- 可选择性地做重排序（Re-rank），把最相关的排到前面；
- 把这些片段作为上下文返回给 `chat_service`。

**为什么要用 RAG？** 因为大模型本身不知道你的专属知识（比如某本医学手册、某个人物的私密日记），通过检索，我们能把相关知识“塞”进提示词里，让模型基于这些信息回答，而不是胡编乱造。

### 3. `llm_service.py` —— 大模型调用

这个服务封装了对大模型 API 的调用。它负责：

- 管理 API Key（从环境变量读取）；
- 设置模型参数，比如温度（Temperature，控制回答的创造性）、最大 token 数；
- 实际发送请求并解析响应。

目前系统支持 OpenAI 兼容的接口，后续可以无缝替换成国产模型或者本地部署的模型。

### 4. `memory_service.py` —— 对话记忆

这个服务解决的是“模型会忘记之前说过什么”的问题。它负责：

- 维护每个会话的对话历史；
- 当对话轮次超过阈值时，自动截断或压缩历史，避免提示词过长；
- 保证模型能记得几分钟前说过的话。

### 5. `knowledge_service.py` 与 `pdf_ingest_service.py` —— 知识库处理

当你上传一份 PDF 时，`knowledge_service` 会调用 `pdf_ingest_service` 来解析 PDF、提取纯文本，然后分块（Chunking），再调用向量化服务存入向量库。这个过程叫“知识库的离线索引”。

**小结：** 业务逻辑层把所有复杂的能力封装成独立的服务，各司其职，互相协作。如果要优化系统的回答质量，主要就是改这一层的代码。

## 四、第三层：数据访问层与数据库层 —— 系统的“仓库”（1.5分钟）

**对应目录：`app/repositories/` 和 `app/db/`**

我把这两层放在一起讲，因为它们紧密配合，负责数据的持久化存储。

### `repositories/` —— 数据访问层

这一层封装了所有数据库操作。比如：

- `character_repository.py` 提供创建角色、查询角色、删除角色的函数；
- `conversation_repository.py` 负责保存每一条对话记录、查询历史会话；
- `user_repository.py` 管理用户注册、登录验证。

**设计意图：** 把“怎么存数据”和“业务逻辑”分离开。如果以后我换数据库（从 SQLite 换到 PostgreSQL），只需要改这一层，上层的业务逻辑完全不用动。

### `db/` —— 数据库模型层

- `models.py` 定义了数据库表结构：用户表、角色表、对话记录表、知识库元数据表；
- `session.py` 管理数据库连接池和事务；
- `init_db.py` 是初始化脚本，第一次运行时会自动创建表并插入默认角色。

**数据库设计与 RAG 的关系：** 对话记录表会保存用户和 AI 的每一条消息，这是实现多轮对话记忆的基础。知识库表记录每个上传文档的元数据（文件名、上传时间、分块数量等）。

## 五、第四层：核心配置与公共组件 —— 系统的“地基”（1分钟）

**对应目录：`app/core/`**

这一层为整个系统提供基础支撑：

- **`config.py`**：读取 `.env` 文件里的配置项，比如 OpenAI API Key、数据库地址、JWT 密钥。这样敏感信息不会硬编码在代码里。
- **`security.py`**：提供密码加密（bcrypt）、JWT 生成和校验功能，保证用户登录安全。
- **`deps.py`**：依赖注入模块，比如在接口层需要获取当前登录用户信息时，通过 `get_current_user` 这个依赖函数自动完成 Token 解析和用户查询。
- **`blocked_words.py`**：敏感词过滤配置，可以对用户输入或模型输出做内容风控，保证系统合规。

**这一层的特点是：** 几乎所有上层模块都会用到它，属于“基础设施”。

## 七、总结与优化方向（1分钟）

### 总结

这个项目的核心优势在于：

1. **架构清晰**：接口层、业务层、数据层分离，符合软件工程最佳实践；
2. **功能完整**：从用户登录、角色管理、对话交互到知识库上传，形成闭环；
3. **可扩展性强**：RAG 检索策略可以灵活调整，大模型可以随时替换；
4. **工程化完善**：包含单元测试、接口测试、性能测试，以及完整的项目文档。

### 后续优化方向

- **检索效果提升**：在 `rag_service.py` 中引入混合检索（向量检索 + 关键词检索）和重排序模型；
- **长记忆优化**：在 `memory_service.py` 中实现对话摘要压缩，支持无限轮次对话；
- **多模态扩展**：支持图片理解能力，用户可以和角色“看图说话”；
- **角色克隆**：上传某人的聊天记录，自动生成一个模拟该人格的角色。
