***\*项目代码说明文档\****

项目名称

***\*- Role-playing system based on RAG_new\*******\*（\*******\*基于RAG的角色扮演对话系统(新)\*******\*）\****

项目目的

构建可扮演多种角色（如医生、律师、虚拟朋友等）的聊天机器人 支持多用户、多角色、多轮对话 具备知识库动态更新与RAG检索能力

 

一、项目整体作用

这是一个“基于 RAG 的角色扮演系统”。

整体基础流程是：

\- 用户登录/注册

\- 选择角色

\- 发送聊天问题

\- 后端根据角色与知识库内容进行检索增强生成（RAG）

\- 前端展示对话结果、历史记录和知识库信息

\- 支持本地启动，也支持通过内网穿透，让外部用户访问

当前项目已经改成“单入口运行”思路：

\- 执行根目录 `main.py`

\- 后端 FastAPI 运行在 `8000`

\- 前端生产页面由后端托管（前提是执行过 `frontend` 的构建）

 

 

对话按 `(user_id, character_id)` 维度聚合为会话，消息写入 `chat_message`。

***\*对话按\**** ***\*(user_id, character_id)\**** ***\*维度聚合为会话，消息写入\**** ***\*chat_message\*******\*。\****

***\*user_id\**** = 你的用户 ID（谁在说话）***\*character_id\**** = 你用的机器人 ID（哪个角色）

所有聊天记录都存到数据库的 chat_message（对话信息） 表里

 

1. 文档切分（段落/固定长度）。
2. BGE-m3 向量化（稠密 + 稀疏，由独立推理服务完成）。
3. Milvus 混合检索（稠密 + 稀疏 / BM25 变体，视 Collection 设计而定）。
4. BGE-rerank 重排序。
5. 与 Redis 短期记忆、角色提示词模板拼接上下文。
6. 调用大模型生成回复。

1.运作流程

1. 用户执行 `python main.py`
2. 后端 FastAPI 在 `8000` 启动
3. 如果 `frontend/dist` 存在，前端页面由后端直接托管
4. 用户通过浏览器访问页面
5. 前端调用 `/api/...` 接口
6. 后端根据路由进入认证、聊天、知识库等业务
7. 服务层调用仓储层、RAG、LLM、数据库
8. 返回结果给前端展示
9. 终端打印请求日志和状态码

 

 

2. 技术栈

| 层级 | 技术 |

| --- | --- |

| Web 框架 | FastAPI |

| 关系库 | MySQL |

| 缓存 | Redis |

| 向量库 | Milvus（`pymilvus` 懒加载）--【用到时才加载，不用不加载】 |

{

正常加载：程序一启动 → 立刻加载 pymilvus 库 → 占用内存

懒加载：  程序一启动 → 不加载 pymilvus

 你用到 Milvus 向量库时 → 才临时加载

}

如果一上来就加载 pymilvus，会：变慢、占内存，可能会报错

· 如果你不用 Milvus：完全不用管，不影响你

· 如果你要用 Milvus：正常安装 pymilvus 即可，用到时会自动加载

· 好处：启动更快、占内存更小、不装也不报错

 

 

| 嵌入模型/重排模型 | BGE-m3 （把句子 / 文档 / 问题变成向量（Embedding）/ BGE-rerank（检索出来的结果太乱，它负责重新排序） |这是给文本做向量、以及给检索结果做精准排序的模型

| 大模型 | 目前本项目使用的本地Ollama-deepseek-r1:7b,它可以调用在线 API 如deepseek、硅基、千问等在线大模型|

本地好处：可以简单调用、不用钱

在线优点：可以让回答更流畅、更像‘智能体’，返回检索更快

\## 4. 数据库设计（逻辑）

`user`：账号凭证、昵称、`preferred_character_ids`（JSON 文本）。

`character`：角色定义、提示词模板、`knowledge_base_id`。

`conversation`：`(user_id, character_id)` 唯一会话（应用层 get_or_create），`updated_at` 在消息写入时刷新。

`chat_message`：一问一答。

`knowledge_document`：上传文件元数据与索引状态。
